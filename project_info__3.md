# crmks — Strong & Weak Sides (Explore Findings)

## Summary
`crmks` — это веб‑приложение (Vue 3 + Pinia + Vue Router) с FastAPI backend, которое управляет сущностями CRM/плана: задачи, leads, клиенты/контакты, предложения (с email), календарь, KPI‑планы и т.д., плюс отдельный AI‑endpoint для “чат‑агента”, который может вызывать серверные tools для работы с доменной моделью. Backend при старте поднимает БД (PostgreSQL async либо SQLite fallback), подключает Redis token store и запускает очередь/фоновые задачи в зависимости от доступности Postgres.

## Architecture (что важно понимать)
- **Фронтенд:** SPA на Vue 3, состояние хранится в **Pinia stores** (например, `src/stores/tasks.ts`), доступ к API — через обертки `src/api/*`.
- **Backend:** FastAPI в стиле **router → service**: `backend/main.py` создаёт приложение, в `lifespan` делает инициализацию (DB/queue/Redis tokens), затем `register_routes(app)` регистрирует доменные роуты.
- **DB стратегия:** код рассчитан на две среды — Postgres (asyncpg/pool) и SQLite (синхронный fallback). Это существенно влияет на поведение: в одном режиме запросы и конкуренция “async”, в другом — “sync”.
- **Auth:** сессии по `Bearer token`, токены хранятся в Redis и обновляются скользящей TTL логикой (sliding expiration). AI‑маршрут использует “lazy” auth‑механизм, чтобы избежать циклов/проблем с dependency graph.

## Сильные стороны (что реально “хорошо”)

### 1) Чёткая модульность backend (router/service)
В `crmks/backend/main.py` и далее в структуре проекта видно разделение:
- **routers** отвечают за HTTP/валидацию и маршрутизацию,
- **services** — за бизнес‑логику и работу с БД.
**Почему это сильное:** проще тестировать/менять доменную логику без переписывания контроллеров, и проще прослеживать “какой слой сломался”.

### 2) Встроенная поддержка “локальная разработка vs прод”
В `crmks/backend/main.py` на старте делаются gated‑инициализации:
- `startup_init_db()` (schema/seed),
- очередь через `init_queue_manager()` (по сути включается только при подходящей среде),
- async pool и Redis токены.
**Почему это сильное:** можно запускать локально без полного прод‑стека (или с частичным), и при этом не терять общий контракт API.

### 3) Сессии через Redis + sliding TTL
Токены инициируются и закрываются через lifespan (`init_token_store`, `close_token_store`). На уровне логики auth токен валидируется и TTL “продлевается”.
**Почему это сильное:** пользовательский UX корректный (активная работа не обрывается), а сервер остаётся источником истины по сессиям (не “самописные cookie‑симуляторы” на фронте).

### 4) Dual‑формат AI запросов (миграционно‑устойчивый API)
`agentApi.parseLeads` показывает стиль API на фронте; отдельно backend AI‑route проектирован под “новый” и “legacy” формат запроса.
**Почему это сильное:** снижается стоимость миграции клиентского формата и уменьшается риск “сломали фронт — всё остановилось”.

### 5) Нормальная композиция фронтенда через Pinia + API clients
В `crmks/src/stores/tasks.ts` CRUD реализован аккуратно:
- `list()` тянет список и обновляет store,
- `create()` дописывает локально `unshift`,
- `update()` делает in‑place `Object.assign`,
- `remove()` фильтрует локально.
**Почему это сильное:** UI обновляется сразу и предсказуемо, без лишних лишних перезагрузок.

## Слабые стороны (риски для разработки/поддержки)

### 1) Дублирование логики auth (особенно для AI)
Есть два “пути” auth: dependency auth для обычных роутов и отдельный “lazy auth” внутри AI маршрута. Это подтверждается тем, что AI‑маршрут делает собственную реализацию шагов Redis+DB.
**Риск:** со временем разные пути могут начать вести себя по‑разному (TTL refresh, ошибки, формат ответов, edge cases).
**Типичный симптом:** “в обычных API всё работает, а в AI чат иногда 401/не обновляется сессия”.

### 2) Schema/seed выполняется при каждом старте приложения
`startup_init_db()` вызывается в `lifespan` при каждом запуске.
**Риск:**
- медленнее старты,
- возможные расхождения “ожидаемой” схемы и фактических миграций,
- в прод‑нагрузке это неприятно (особенно если много экземпляров стартует параллельно).
**Почему это важно:** миграции и seed лучше делать отдельно/одноразово, а не в “runtime startup”.

### 3) Фрагильная SQL‑адаптация (диалекты через строковые замены)
Ранее в документации к проекту отмечено, что SQL адаптируется по тексту (замены `%s`, `ILIKE`, удаление `RETURNING id` для SQLite).
**Риск:** небольшое изменение SQL в одном месте может сломать адаптацию “тихо”, без compile‑time защиты.
**Вероятные последствия:** “в Postgres работает, в SQLite — нет”, или наоборот, “выборка возвращает не те поля/порядок”.

### 4) Зависимость от порядка колонок при маппинге строк БД
Для mapping в задачах/DTO отмечался подход с фиксированными индексами (например, `r[9]`, `r[13]`).
**Риск:** изменение SELECT (добавили поле, поменяли порядок) ломает маппинг без ошибок типов.
**Что делать разработчику:** если трогать SELECT — обязательно синхронизировать mapping.

### 5) Открытый CORS (`allow_origins=["*"]`)
`crmks/backend/main.py` включает CORS полностью.
**Риск безопасности:** при наличии bearer‑token логики и неправильной защите на уровне фронта/хостинга можно получить нежелательные сценарии.
**Примечание:** безопасность всё равно зависит от того, как организован хранение токена (localStorage vs memory vs cookie), но CORS‑широкость — это потенциальный рычаг для атак.

### 6) Разный concurrency/performance профиль в двух DB режимах
Postgres режим — async, SQLite fallback — sync. Даже если API единый, поведение under load может быть радикально различным.
**Риск:** тесты “на SQLite” могут не ловить реальные гонки/тайминги из Postgres.

## Developer-facing: что нужно знать, чтобы работать эффективно

### Инварианты, которые важно не нарушать
1) **Контракт API**: фронтенд ожидает JSON‑формы из `src/api/*`; backend меняет DTO — фронт может “молча” сломаться.
2) **Auth refresh**: если вы добавляете новый endpoint, используйте стандартный dependency auth для консистентности, а не копируйте AI‑lazy‑подход.
3) **DB режим**: любая правка в `services/*` может затронуть обе ветки (Postgres vs SQLite). Проверяйте обе логики.

### Где чаще всего “болит”
- Любые изменения SQL в `services/*` (особенно с mapping по индексам и string‑adaptation).
- Любые изменения в auth (особенно в AI чат: легко получить дрейф поведения).
- Любые изменения в схеме: потому что схема/seed стартуют при запуске runtime.

## Рекомендации (приоритеты)
1) **Унифицировать auth для AI и обычных роутов** (один источник истины: TTL refresh + формат ошибок + загрузка user).
2) **Перевести migration/seed на явный процесс** (или хотя бы добавить “одиночное выполнение”/lock).
3) **Заменить string-based SQL adaptation на более надёжный слой** (например, query builder или явные отдельные запросы для SQLite/Postgres).
4) **Уйти от mapping по индексам** к mapping по именам колонок (named rows) или явным DTO‑конверторам.
5) **Сузить CORS по production‑origin** и документировать безопасную схему хранения токена.

## Коротко: “профит” и “тормоза”
- **Профит:** хорошая модульность (router/service), понятная авторизация через Redis tokens, адекватная структура фронта (Pinia + store updates), AI API учитывает migration/legacy.
- **Тормоза:** дублирующий auth, startup‑side schema/seed, хрупкая SQL‑адаптация и mapping по индексам, plus security/ops риски (CORS wide) и различия поведения в двух DB режимах.
